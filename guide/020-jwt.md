# JSON Web Token (JWT) Authentication

JSON Web Tokens are an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.

---

## 1. Prerequisites & Conceptual Basics

### Stateful vs. Stateless Authentication
- **Stateful (Sessions)**: The server stores session records in memory or a database, and gives the client a session ID cookie. To validate subsequent requests, the server must query the session store, creating performance bottlenecks at scale.
- **Stateless (JWT)**: When a user logs in, the server generates a cryptographically signed token containing user information and sends it to the client. The server does not store the token. Instead, on subsequent requests, the server validates the token's cryptographic signature, making it highly scalable for distributed microservices.

### JWT Structure
A JWT consists of three parts separated by dots (`.`):
`Header.Payload.Signature`
1. **Header**: Contains the token type (JWT) and signing algorithm (e.g. HS256).
2. **Payload**: Contains the claims (e.g. user ID, email, role).
3. **Signature**: Created by combining the encoded header, encoded payload, and your private secret key, then hashing them together. This verifies that the token hasn't been tampered with.

---

## 2. Theory & Deep Dive

### Token Expiration Lifecycle
Tokens must have a defined expiration time (e.g. `expiresIn: '15m'`). If a token does not expire, an attacker who steals it can access the system indefinitely. To maintain sessions without forcing users to re-login repeatedly, use a **Refresh Token** architecture:
- Store a short-lived Access Token in client memory.
- Store a long-lived Refresh Token in a secure, HTTP-only cookie to request new access tokens.

## 3. Code Implementation

Install the dependencies first: `npm install jsonwebtoken bcrypt cookie-parser`

### File: `middleware/auth.js` (JWT Verification Middleware)
```javascript
const jwt = require("jsonwebtoken");

// Middleware to protect routes by validating incoming JWT cookies
async function protect(req, res, next) {
  try {
    let token = req.cookies.token;

    // Check if token exists
    if (!token) {
      return res.status(401).json({ success: false, message: "Access Denied: Please log in." });
    }

    // Verify token signature with secret key
    const decoded = jwt.verify(token, process.env.JWT_SECRET || "fallback_secret_key");
    
    // Attach decoded token payload (e.g., user ID) to the request object
    req.user = decoded;
    
    next();
  } catch (error) {
    res.status(401).json({ success: false, message: "Unauthorized: Invalid or expired token." });
  }
}

module.exports = { protect };
```

## 4. Self-Contained Mini-Project: Secure User Authentication Hub

We will build an Express authentication server with user registration (using bcrypt hashing), login (generating signed JWT cookies), and protected routes.

### Project Setup
```text
express-jwt-auth/
├── server.js
└── package.json (requires: express, jsonwebtoken, bcrypt, cookie-parser)
```

### File: `server.js`
```javascript
const express = require("express");
const jwt = require("jsonwebtoken");
const bcrypt = require("bcrypt");
const cookieParser = require("cookie-parser");
const app = express();

app.use(express.json());
app.use(cookieParser());

const JWT_SECRET = "super-secret-production-hash-key-88";

// In-memory Database
const users = [];

// Middleware protecting routes
function authGuard(req, res, next) {
  const token = req.cookies.authToken;
  if (!token) {
    return res.status(401).json({ error: "Access denied. Login required." });
  }
  try {
    const verified = jwt.verify(token, JWT_SECRET);
    req.user = verified;
    next();
  } catch (err) {
    res.status(400).json({ error: "Invalid credentials token." });
  }
}

// 1. REGISTER Route
app.post("/api/auth/register", async (req, res) => {
  try {
    const { username, password } = req.body;
    if (!username || !password) {
      return res.status(400).json({ error: "Username and password required." });
    }

    const salt = await bcrypt.genSalt(10);
    const hashedPassword = await bcrypt.hash(password, salt);

    users.push({ username, password: hashedPassword });
    res.status(201).json({ success: true, message: "User registered successfully." });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// 2. LOGIN Route
app.post("/api/auth/login", async (req, res) => {
  try {
    const { username, password } = req.body;
    const user = users.find(u => u.username === username);

    if (!user || !(await bcrypt.compare(password, user.password))) {
      return res.status(401).json({ error: "Invalid username or password." });
    }

    // Sign JWT
    const token = jwt.sign({ username: user.username }, JWT_SECRET, { expiresIn: "1h" });

    // Store in cookie
    res.cookie("authToken", token, { httpOnly: true, maxAge: 3600000 });
    res.json({ success: true, message: "Login successful!" });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// 3. SECURE profile route
app.get("/api/profile", authGuard, (req, res) => {
  res.json({ success: true, username: req.user.username, secretData: "Stateless security achieved!" });
});

app.listen(3000, () => console.log("Auth hub running at http://localhost:3000"));
```

## 5. Advanced Production Practices & Security

### Token Blacklisting (Revocation)
Since JWT authentication is stateless, the server cannot invalidate a token once it is issued. If a user logs out or their account is compromised, the token remains valid until it expires.
- **Production Solution**: Use a fast, in-memory store like **Redis** to maintain a blacklist of revoked tokens. When a user logs out, add their token to the Redis blacklist with a TTL matching the token's remaining lifespan. Your authentication middleware should check this blacklist on every request.

---

## 6. Key Takeaways
1. A JWT is signed, not encrypted. Anyone can decode the base64-encoded token to read its contents. **Never store sensitive data (like passwords, keys, or SSNs) in the JWT payload.**
2. Set the `httpOnly` flag to `true` when storing JWTs in cookies. This prevents malicious client-side JavaScript (XSS attacks) from reading the token.
3. The `JWT_SECRET` must be kept extremely secure. If it is leaked, attackers can generate valid signatures and compromise the entire system.
