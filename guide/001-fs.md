# Node.js File System (`fs`) Module

The Node.js File System (`fs`) module enables you to interact with the file system on your operating system. It provides methods to create, read, update, delete, and rename files or directories.

---

## 1. Prerequisites & Conceptual Basics

### Theoretical Basics
Every operating system stores data in files grouped into directories. To read or write these files, programming languages use "system calls" to request permission from the operating system kernel.
In standard synchronous programming (like Python or PHP by default), when you read a file, execution blocks (stops) until the file is loaded from physical storage (SSD/HDD) into RAM. Since I/O (Input/Output) operations are thousands of times slower than CPU operations, blocking is highly inefficient for web servers.

Node.js solves this using its **Asynchronous Event-Driven Architecture**. When Node.js performs a file operation:
1. It delegates the physical read/write request to the operating system or a background pool of workers managed by **Libuv**.
2. Node.js continues executing subsequent lines of code.
3. When the file transfer completes, Libuv triggers an event that inserts a callback, resolves a promise, or continues the async function inside the main event loop thread.

---

## 2. Theory & Deep Dive

### Core Approaches
Node.js offers three ways to work with the file system:
- **Synchronous**: Blocks execution until the operation completes. Useful for initialization tasks (e.g., loading config files at startup).
- **Callback (Asynchronous)**: Non-blocking, executes a callback function when done.
- **Promises (Asynchronous)**: Modern non-blocking API that works with `async/await` (Recommended for clean, scalable code).

### Comparison: Sync vs Callbacks vs Promises
| Approach | Syntax Complexity | Non-blocking? | Error Handling Style |
| :--- | :--- | :--- | :--- |
| `fs.readFileSync` | Simple, linear | No (Blocks!) | `try-catch` |
| `fs.readFile` | Callback hell (nested) | Yes | Error-first parameters (`err, data`) |
| `fs/promises` | Clean, linear | Yes | `try-catch` with `async/await` |

## 3. Basic Code Implementation

Here is how you use the promise-based `fs` module in CommonJS style:

```javascript
const fs = require("fs/promises");
const path = require("path");

// Define file path
const filePath = path.join(__dirname, "example.txt");

// 1. Writing a File Asynchronously
async function writeExample() {
  try {
    const data = "Hello, this is a sample text file written using Node.js fs module!";
    // fs.writeFile completely overwrites the file if it exists
    await fs.writeFile(filePath, data, "utf-8");
    console.log("File written successfully!");
  } catch (error) {
    console.error("Error writing file:", error.message);
  }
}

// 2. Reading a File Asynchronously
async function readExample() {
  try {
    // Reading file content
    const content = await fs.readFile(filePath, "utf-8");
    console.log("File Content:", content);
  } catch (error) {
    console.error("Error reading file:", error.message);
  }
}

// 3. Appending to a File Asynchronously
async function appendExample() {
  try {
    const extraData = "\nThis line has been appended to the existing content.";
    await fs.appendFile(filePath, extraData, "utf-8");
    console.log("Data appended successfully!");
  } catch (error) {
    console.error("Error appending to file:", error.message);
  }
}

// 4. Deleting (Unlinking) a File Asynchronously
async function deleteExample() {
  try {
    await fs.unlink(filePath);
    console.log("File deleted successfully!");
  } catch (error) {
    console.error("Error deleting file:", error.message);
  }
}
```

## 4. Self-Contained Mini-Project: Simple JSON Logger database

We will build a console utility that logs server transactions to a JSON array file on disk, behaving like a basic persistent database logger.

### Project Setup
Create a folder structure:
```text
json-logger/
├── logger.js
└── logs.json (created automatically)
```

### File: `logger.js`
```javascript
const fs = require("fs/promises");
const path = require("path");

const logFilePath = path.join(__dirname, "logs.json");

// Core database writer
async function writeLog(actionName, details) {
  try {
    let logsList = [];
    
    // Check if the log file already exists
    try {
      const rawData = await fs.readFile(logFilePath, "utf-8");
      logsList = JSON.parse(rawData);
    } catch (readError) {
      // If file doesn't exist, start with an empty list
      if (readError.code !== 'ENOENT') {
        throw readError; // Re-throw other unexpected errors
      }
    }

    // Append new log object
    const newLog = {
      timestamp: new Date().toISOString(),
      action: actionName,
      details: details
    };
    logsList.push(newLog);

    // Save logs back to disk formatted neatly (indentation of 2 spaces)
    await fs.writeFile(logFilePath, JSON.stringify(logsList, null, 2), "utf-8");
    console.log(`[INFO] Transaction Logged: ${actionName}`);
  } catch (error) {
    console.error("[ERROR] Failed to write log:", error.message);
  }
}

// Execute dummy transactions
async function startApp() {
  await writeLog("USER_SIGNUP", { email: "student@example.com", role: "student" });
  await writeLog("PASSWORD_CHANGE", { email: "student@example.com" });
  
  // Read and print out final file status
  const finalContent = await fs.readFile(logFilePath, "utf-8");
  console.log("\nSaved Logs DB Output:\n", finalContent);
}

startApp();
```

## 5. Advanced Production Practices & Security

### File descriptor leaks
Every time you open or read a file using low-level tools, a system handler called a "file descriptor" is allocated. In massive applications, if you open streams or file handles without closing them, the server runs out of file descriptors, causing it to crash and block new client requests. Always clean up resource streams.

### Stream-based uploads vs. Memory load
Using `fs.readFile` loads the **entire** file into RAM memory. If 100 users try to read 500MB video files simultaneously, the server requires 50GB of RAM and will crash instantly. Use Node.js **Streams** (`fs.createReadStream`) to process files chunk-by-chunk without flooding memory.

### Security
1. **Directory Traversal Attacks**: If your code accepts user-provided filenames (like `req.query.file`), malicious users can pass paths like `../../etc/passwd` to read system configuration secrets. Always use `path.basename()` or sanitize inputs.
2. Avoid setting directories to `777` permissions (fully readable/writable/executable by everyone). Keep permissions restricted (like `755` for directories and `644` for files).

---

## 6. Key Takeaways
1. **Always handle errors** using `try-catch` blocks to prevent your Node.js application from crashing unexpectedly.
2. Prefer **promise-based asynchronous methods** (`require("fs/promises")`) over synchronous functions for any operations run in request handlers, as blocking the event loop halts response times for other active users.
3. Use the **`path` module** (`path.join`) to resolve file paths dynamically, ensuring cross-platform compatibility (Windows uses `\`, macOS/Linux uses `/`).
